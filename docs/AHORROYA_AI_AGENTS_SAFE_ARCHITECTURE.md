# AhorroYA AI Agents Safe Architecture

Date: 2026-05-10

## Status

- `ENABLE_AI_AGENTS=false` by default.
- `ENABLE_ADMIN_AI_PANEL=false` by default.
- `ENABLE_AGENT_SCHEDULER=false` by default.
- `AI_AUTONOMY_LEVEL=LEVEL_0_READ_ONLY` by default.
- `AI_PROVIDER=mock` by default.
- Production agent endpoint is disabled by code.
- `AI_KILL_SWITCH=true` immediately blocks the agent endpoint.

## Security Boundary

AhorroYA agents may analyze, report, and propose. They must not directly mutate production systems.

Allowed agent work:

- Analyze metrics.
- Detect errors.
- Suggest improvements.
- Generate reports.
- Propose code, docs, roadmap, and operational tasks.
- Create prioritized task records.

Forbidden agent work:

- Touch production directly.
- Modify Vercel Production envs.
- Activate PayPal live.
- Delete data.
- Disable RLS.
- Print secrets.
- Auto-approve critical changes.
- Deploy production without release gates.

## Data Model

| Table | Purpose | Write Access |
| --- | --- | --- |
| `agent_tasks` | Queue proposed or scheduled agent work | `admin`, `internal_job` |
| `agent_logs` | Audit events and blocked actions | `admin`, `internal_job` |
| `agent_reports` | Structured reports generated by agents | `admin`, `internal_job` |
| `agent_suggestions` | Proposed product/security/ops changes | `admin`, `internal_job` |
| `agent_executions` | Execution metadata and summaries | `admin`, `internal_job` |
| `agent_memory` | Agent state and compact memory | `admin`, `internal_job` |

All tables are created with RLS enabled. Policies use `agent_authorized_role()` and accept only JWT `app_metadata.role` values `admin` and `internal_job`.

## Role Model

Authorization must use `auth.jwt()->app_metadata->role`.

Do not use `user_metadata` for authorization. User metadata is user-editable and unsafe for RLS decisions.

Authorized roles:

- `admin`: human operator with administrative access.
- `internal_job`: controlled backend/job identity.

Unauthorized roles:

- `anon`
- `authenticated`
- `user`
- `merchant`
- `moderator`

## Runtime Gates

The API route `/api/v1/ai/agents` enforces:

1. `AI_KILL_SWITCH=true` returns `ai_agents_kill_switch_active`.
2. `APP_ENV=production` returns `ai_agents_production_disabled`.
3. `ENABLE_ADMIN_AI_PANEL=false` returns `admin_ai_panel_disabled`.
4. Caller must have `admin` or `internal_job`.
5. `run` and `runAllSafe` require `ENABLE_AI_AGENTS=true`.
6. Level 4 autonomy requires `ENABLE_AI_LEVEL4_OVERRIDE=true`, but this is blocked for staging/initial production checks.

## Persistence

`SupabaseAgentMemory` stores sanitized summaries through Supabase REST using the server-only service role key.

Sanitization:

- JSON keys matching secret/token/key/password/credential are redacted.
- Large strings are truncated.
- Failed persistence falls back to in-memory storage and records a warning.

## Production Position

Production is disabled for agents. The initial production posture is:

```text
AI_PROVIDER=mock
AI_GATEWAY_ENABLED=false
AI_KILL_SWITCH=false
ENABLE_AI_AGENTS=false
ENABLE_AGENT_SCHEDULER=false
ENABLE_ADMIN_AI_PANEL=false
ENABLE_AI_LEVEL4_OVERRIDE=false
AI_AUTONOMY_LEVEL=LEVEL_0_READ_ONLY
```

To emergency-disable every agent surface:

```text
AI_KILL_SWITCH=true
```

## Migration Policy

The existing agent migration is safe to keep versioned. Do not apply migrations to production automatically. Production application requires:

- SQL backup.
- Revert plan.
- RLS verification.
- Production release gate.
- No secrets in logs.
